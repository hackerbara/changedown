Agent sends propose_change() via MCP
2
ChangeTracks engine creates CriticMarkup + footnote
3
Adapter translates CriticMarkup → proofSuggestion marks
4
Adapter calls ydoc.transact() with suggestion marks (not direct edits)
5
Broadcasts suggestions to collab clients
6
Human reviews in Milkdown UI — accept/reject
7
Adapter translates decision back → CriticMarkup status update

Key Architectural Question

The fundamental tension: who is the source of truth?

In Proof's world, the Y.Doc is authoritative and markdown is derived. In ChangeTracks' world, the markdown file is authoritative and everything else is derived. The adapter must mediate this inversion — and the design decision of which side "wins" during conflicts shapes every other choice.
