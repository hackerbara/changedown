Breakpoint #1: remarkProofMarks() doesn't recognize CriticMarkup

Proof's remark plugin only matches <span data-proof="..."> HTML. CriticMarkup delimiters like {++text++} pass through as literal text nodes. Without a bridge plugin, CriticMarkup becomes dead characters in the ProseMirror tree.

Breakpoint #2: proofMarkHandler() serializes to HTML spans, not CriticMarkup

Even if we get CriticMarkup INTO Proof as proofSuggestion marks, when Proof serializes back to markdown it outputs <span data-proof="suggestion"> — not CriticMarkup syntax. The roundtrip destroys our format.

Solution: Two-way remark transform (@changetracks/remark)

On parse: Runs BEFORE remarkProofMarks(). Finds CriticMarkup patterns in text nodes, converts to proofMark MDAST nodes that Proof's pipeline already understands.
On serialize: Runs AFTER proofMarkHandler(). Intercepts proofMark nodes and emits CriticMarkup syntax instead of HTML spans.
Net effect: CriticMarkup flows through Proof's existing mark infrastructure without Proof needing any code changes.

Breakpoint #3: Highlight ({==text==}) has no clean Proof equivalent

Proof has proofFlagged (human review marker) and proofApproved, but neither maps to "highlighted for attention." Options: (a) map to proofFlagged with a custom attr, (b) extend Proof's mark schema, (c) accept lossy mapping. This is a semantic gap, not a technical one.

Verdict: 4/5 types map cleanly. Highlight is a minor gap.

Insertion, deletion, substitution, and comment have direct 1:1 equivalents. Highlight can map to proofFlagged with a source: 'criticmarkup' attribute to distinguish. No schema changes required in Proof — proofFlagged already supports arbitrary data attrs.

The Triple-Write Test: ydoc.transact() Atomicity

Proof's canonical mutation updates three Yjs shared types atomically. Does CriticMarkup survive all three?

Yjs Shared Type
Verdict
Analysis
1

Y.XmlFragment ('prosemirror')

ProseMirror document tree. Written via replaceYXmlFragment() which calls prosemirrorToYXmlFragment(pmDoc, fragment).

PASS
If the remark plugin converts CriticMarkup → proofMark MDAST nodes, Milkdown's schema runner converts them to ProseMirror proofSuggestion marks. y-prosemirror handles marks natively — they serialize to Y.XmlFragment as attributes on Y.XmlText elements. No special handling needed.
2

Y.Text ('markdown')

Raw markdown string. Written via applyYTextDiff() which finds longest common prefix/suffix and does delete+insert.

DEPENDS
Question: What gets written to Y.Text? If the remark serializer converts proofMark back to CriticMarkup (via our plugin), Y.Text contains CriticMarkup. If it uses Proof's default proofMarkHandler(), Y.Text contains <span data-proof> HTML. This is where the roundtrip format is decided.
3

Y.Map ('marks')

Metadata sidecar. Written via applyMarksMapDiff() which does key-level diffing of mark objects.

NEEDS BRIDGE
This is where CT footnote metadata goes. Proof's marks have: id, kind, by, at, quote, data. CT footnotes have: author, date, status, approvals[], discussion[], revisions[]. The bridge must serialize CT metadata into Proof's data field. Proof's mark schema allows arbitrary data payloads, so this is mechanically possible.

Edge Case Stress Tests

Scenario
Verdict
Analysis
A

Block boundary crossing

{++text\n\nMore text++} — insertion spans two paragraphs

BREAKS
ProseMirror marks cannot span block nodes. A mark on a paragraph cannot extend into the next paragraph. Proof's proofSuggestion is an inline mark (spanning: true but only within a block). Remark would split this across two paragraph nodes. The remark plugin must either reject cross-block CriticMarkup or split into per-block segments with a shared group ID.
B

Substitution content storage

{~~old~>new~~} — where does "old" text live in ProseMirror?

PASS
Proof's proofSuggestion(kind:'replace') stores the replacement content in the mark's content attribute, while the original text remains in the document. This matches CriticMarkup's model exactly — the "old" text is the marked content, "new" is in the mark attr. Bidirectional mapping is clean.
C

Concurrent CRDT merge with pending CriticMarkup

Two Yjs clients: one inserts CriticMarkup-derived marks, another edits nearby text

PASS
Once CriticMarkup is converted to proofSuggestion marks, it's just a ProseMirror mark in Yjs. y-prosemirror handles mark merging natively via Y.XmlText attributes. The CRDT merge is format-agnostic — it doesn't know or care that the mark originated from CriticMarkup.
D

Footnote reference survival

[^ct-4] in markdown — does remark-footnotes consume it?

ORDERING
If remark-footnotes runs before our plugin, it consumes [^ct-4] as a standard footnote and removes it from the AST. Our remark plugin must run first to capture CT footnote refs and convert them to mark metadata before remark-footnotes sees them. Plugin ordering in unified is explicit via .use() chain position.
E

suggestions plugin double-wrapping

CT-derived marks arrive via Yjs sync — does the plugin re-wrap them?

PASS
The suggestions plugin explicitly checks tr.getMeta('y-sync$') and passes through unchanged. Remote Yjs changes (including our CT-derived marks) skip the interception logic entirely. No double-wrapping risk.
F

HTML fallback cascade

parseMarkdownWithHtmlFallback() strips HTML on parse failure — affects CriticMarkup?

PASS
The fallback strips <span> and <code> HTML tags. CriticMarkup uses {++/{--/{~~ delimiters, not HTML. The fallback cascade won't touch them. However, if our remark plugin runs first and converts CM to proofMark nodes, the fallback is never needed for our content.
G

Headless server parse

milkdown-headless.ts creates server-side editor — does it load our plugin?

CONFIG
The headless parser builds its own unified pipeline: remarkParse + remarkGfm + remarkProofMarks. Our remark plugin must be injected here too — it's not automatic. The headless builder would need .use(remarkCriticMarkup) added before .use(remarkProofMarks). This is a configuration change, not architectural.
H

CT metadata richness vs Proof mark attrs

Footnote: threaded discussion (depth N), amendment history (r1..rN), approval chains, resolution

LOSSY?
Proof marks have data: MarkData (arbitrary JSON). Structurally it fits. But Proof's UI won't render CT's threaded discussions, amendment chains, or resolution markers — it only understands its own mark kinds. The metadata roundtrips through Y.Map intact but is opaque to Proof's editor. It survives for file export but is invisible during editing.
